## Идеи
- access-логи для режима serve; сейчас логов никаких нет по запроса
- Rootless-контейнер. Сейчас собирается один Docker-образ на базе alpine:3.21 — он уже запускается как non-root (uid 65534) через securityContext в Helm-чарте, но сам образ содержит root filesystem и пакетный менеджер (apk). Нужно собирать второй образ (тег с суффиксом -rootless или отдельный Dockerfile) на базе scratch или distroless, содержащий только статический бинарник и CA-сертификаты, без shell, пакетного менеджера и вообще каких-либо лишних файлов — это минимизирует attack surface и соответствует требованиям к rootless/distroless контейнерам. В Helm-чарте values.yaml нужно по умолчанию переключить image.repository (или добавить вариант) на rootless-образ, сохранив возможность использовать текущий alpine-based образ через override.
- Команда config edit. Сейчас для редактирования конфига вручную пользователь должен сам найти путь к файлу (через config show) и открыть его в редакторе. Нужна команда express-botx config edit, которая по аналогии с kubectl edit открывает конфиг-файл в $EDITOR (с fallback на vi), даёт пользователю отредактировать его, а после сохранения и закрытия редактора — валидирует изменённый YAML (парсинг + проверка обязательных полей) и либо принимает изменения, либо предлагает повторить редактирование или откатиться к исходной версии, если конфиг невалиден. Путь к конфигу определяется через существующий resolveConfigPath / флаг --config.
- Команда config validate. Добавить команду express-botx config validate, которая проверяет синтаксическую корректность конфиг-файла: валидный YAML, известные поля (предупреждение о неизвестных ключах), наличие обязательных полей (host, id, secret/token у ботов), корректность форматов (UUID для id, валидные duration-значения для таймаутов, допустимые значения enum вроде routing_mode), непротиворечивость (например, secret и token одновременно у одного бота, ссылки чатов на несуществующих ботов в multi-bot режиме). Команда не проверяет доступность серверов или валидность токенов — для этого есть bot info. Путь к конфигу — через --config или стандартный resolveConfigPath. Выход: ok при успехе, список ошибок/предупреждений при проблемах, exit code 1 при ошибках.
- Команда bot info --all. Сейчас bot info работает с одним ботом — выбранным через --bot или единственным в конфиге; при multi-bot конфигурации без --bot команда завершается ошибкой. Нужно добавить флаг --all / -A, при котором команда итерирует по всем ботам из конфига (через LoadMinimal + cfg.Bots), для каждого выполняет аутентификацию и выводит сводку (name, host, bot_id, cache, auth status) в виде таблицы (text) или массива (json); при ошибке аутентификации конкретного бота — показывать ошибку в поле auth_status, но продолжать проверку остальных, итоговый exit code ненулевой если хотя бы один бот не прошёл проверку; аналалогично рассмотреть bot ping --all — проверить connectivity всех ботов разом; bot token --all — получить токены всех ботов. Полезно для скриптов, которые обновляют токены или проверяют их валидность; chats list --all — сейчас показывает чаты одного бота (через API). С --all — объединённый список чатов со всех ботов с колонкой bot name. chats import --all — импортировать чаты со всех ботов в конфиг за один вызов.
